## 2. tool / 데이터 스키마 (고정)

프론트/백엔드/기획이 같은 값을 보게 스키마를 먼저 못 박는다.

```
control_device(device, action)
  device : "shade" | "window" | "irrigation"
  action : "open" | "close"      # shade, window
           "on" | "off"          # irrigation

read_data(target)
  target : "temperature" | "humidity" | "production"
```

**Mock 반환 형태**
- 성공: `{ "ok": true, "device": "...", "state": "..." }`
- 실패/모호: `{ "ok": false, "reason": "..." }`

## 3. 오작동 확인 피드백 (안전장치, 핸드오프 §8)

- 없는 장비 / 모호한 명령 → Mock 이 `{ok:false, reason}` 반환
- 모델은 그걸 받아 **되묻거나 상태를 알려준다**
  - 예) "그 장비는 없어요." / "무엇을 여닫을까요?" / "차광막은 이미 닫혀 있어요."
- 절대 조용히 실패하지 않는다 (고령 농가 대상 — 확인을 준다)

## 4. 데모 시나리오 (6개)

| # | 입력 | 종류 | 기대 동작 |
|---|---|---|---|
| 1 | 차광막 닫아줘 | 제어 | shade → close |
| 2 | 창문 열어 | 제어 | window → open |
| 3 | 관수 틀어줘 | 제어 | irrigation → on |
| 4 | 지금 온도 몇 도야? | 조회 | temperature |
| 5 | 오늘 생산량 알려줘 | 조회 | production |
| 6 | 창문 열고 온도도 알려줘 | 복합 | window open + temperature |

> 기획팀이 각 항목의 **기대 응답 문장**과 mock 숫자를 채운다. (`todo/PLANNING.md`)
