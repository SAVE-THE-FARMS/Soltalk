
# SolTalk — 개발 핸드오프 문서

> 이 문서는 Claude Code(또는 다른 개발 환경)에서 `soltalk` repo 개발을 시작할 때
> 읽는 컨텍스트 문서입니다. 기획/결정사항은 여기 정리돼 있으니 반복 설명 불필요.

---

## 1. 프로젝트 한 줄 정의

**SolTalk** — 자연어(음성/텍스트) 명령으로 스마트팜(온실) 장비를 제어하고 현황을 조회하는 AI 에이전트.
주체: (주)세이브더팜즈(SAVE THE FARMS).

본질은 **"자연어 → 의도 파악 → IoT 장비 제어/상태 조회"** 앱. 새로운 기술 스택은 아니며,
가치는 **농업 도메인 특화**(고령 농가 UX, 익숙한 채널, 온실 장비 통합)에서 나온다.

---

## 2. 1차 구현 범위 (이번에 만드는 것)

기획서 프로세스 기준 **Step 1 → 2 → 3-A / 3-B → 4 → 5**:

| Step | 내용                                      | 비고                                      |
| ---- | ----------------------------------------- | ----------------------------------------- |
| 1    | 사용자의 자연어 명령/질문 입력 받기       | 챗봇 UI, (선택) STT                       |
| 2    | 의도 파악 — 제어 / 조회 분류 + 개체 추출 | LLM(OpenAI)로 처리                        |
| 3-A  | (제어) IoT 장비에 명령 전달               | **Mock API** — 가상 장비 상태 변경 |
| 3-B  | (조회) 센서/생산 데이터 조회              | **Mock 데이터**                     |
| 4    | 결과를 자연어 문장으로 변환               | LLM 응답 생성                             |
| 5    | 사용자에게 전달                           | 앱/챗봇 응답                              |

**제어 대상 장비 (가상)**: 차광막(shade), 창문(window), 관수(irrigation)
**명령 예시**: "차광막 닫아줘", "창문 열어", "오늘 생산량 알려줘", "지금 온도 몇 도야?"

## 3. 범위 밖 (이번엔 안 함)

- **솔캐스트(Soilcast) 예측 연동 (Step 3-C)** — 2차 확장 과제. 예측 엔진은 구현돼 있으나
  토양 데이터 커버리지 확충 중. **이 repo에는 솔캐스트 코드/데이터 절대 포함하지 않는다.**
- 실물 하드웨어(실제 온실 장비) 연동 — 이후 과제. 지금은 Mock만.

## 4. 확정된 결정사항

- **프로젝트/repo 이름**: `soltalk` (솔캐스트와 `Sol-` 브랜드 패밀리로 엮임)
- **repo 공개 여부**: **공개(public)**. 참여 대학생 2명의 포트폴리오용.
  - 솔캐스트는 어차피 1차 범위 밖이라 공개 repo에 들어갈 민감 코드가 없음 → repo 쪼갤 필요 없음.
  - 솔캐스트는 나중에 붙일 때 **별도 비공개 repo**로 분리.
- **소유권**: repo **owner = 회사 계정**. 학생 2명은 **collaborator**로 초대. (학생 개인 repo에 두지 않는다)
- **라이선스**: 공개 IP 관계 사전 합의 필요. 회사 통제가 필요하면 `All rights reserved` 또는 회사 소유 명시.

## 5. 공개해도 되는 것 / 절대 안 되는 것

**공개 OK**: 음성/챗봇 UI, 의도분류·개체추출 로직, Mock IoT 어댑터, 응답 생성 — 전부.

**공개 금지 (코드 아님, 데이터/키라 놓치기 쉬움)**:

1. **실제 장비 크리덴셜** — IoT 벤더 API 키, 온실 접속 정보 → `.env` + `.gitignore`. 배포 시 환경변수.
2. **실제 농가/생산 데이터** — 시연은 더미 데이터로. 실데이터 섞지 말 것.
3. **솔캐스트 관련 일체** — 엔진, 토양 데이터.

> 공개 repo는 히스토리에 한 번 올라가면 영구히 남는다. 키/데이터는 **처음부터** 커밋 제외.

## 6. 권장 아키텍처

핵심은 **IoT 연동을 인터페이스로 추상화**해서 Mock ↔ 실장비를 갈아끼울 수 있게 하는 것.

```
사용자 입력
   │
   ▼
[NLU / 의도분류]  ── LLM(OpenAI) function calling: intent(control|query) + entity(device, action)
   │
   ├── control ──▶ IoTAdapter.control(device, action)   ┐
   │                                                     ├── 인터페이스
   └── query   ──▶ IoTAdapter.read(target)               ┘
                        │
                        ▼
              [MockIoTAdapter]  ← 이번 범위. 가상 상태 저장/변경
              [RealIoTAdapter]  ← 나중에 비공개 repo에서 주입 (같은 인터페이스)
                        │
                        ▼
              [응답 생성]  ── 처리 결과 → 자연어 문장(LLM)
                        │
                        ▼
                     사용자
```

- 공개 repo는 `IoTAdapter` **인터페이스 + Mock 구현**만 포함.
- 실장비/솔캐스트 구현은 회사가 같은 인터페이스로 비공개에서 주입.
- → 학생 코드는 실장비·솔캐스트를 몰라도 완결 동작.

## 7. Claude Code에서 할 일 (제안 순서)

1. repo 스캐폴딩 — 폴더 구조, `README.md`, `.gitignore`, `.env.example`, 라이선스 파일
2. `IoTAdapter` 인터페이스 + `MockIoTAdapter`(가상 장비 상태) 구현
3. NLU 모듈 — 자연어 → `{intent, device, action}` (LLM 프롬프트)
4. 응답 생성 — 처리 결과 → 자연어
5. 챗봇 UI / 진입점 (텍스트 먼저, STT는 선택)
6. 데모 시나리오 — 위 명령 예시들로 end-to-end 동작 확인

> 기술 스택(언어/프레임워크)은 아직 미확정. Claude Code 시작 시 정할 것.
