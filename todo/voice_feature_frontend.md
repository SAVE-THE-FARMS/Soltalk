# 음성 인식 기능 — 프론트엔드 요구사항 (Realtime API 연동, 옵션 B)

> 채택 방식 변경: 브라우저 Web Speech API(옵션 A) → **OpenAI Realtime API 연동(옵션 B)**
> WebRTC를 통해 브라우저가 OpenAI와 직접 실시간 오디오를 주고받으며, function-call 이벤트를 우리 백엔드로 중계하는 구조.

---

## 1. 필요한 UI 요소

### 1.1 음성 모드 진입/종료 버튼

- 기존 "탭 눌러 말하기(푸시투토크)"가 아니라, **음성 모드를 켜면 지속적인 연결(세션)이 유지**되는 방식으로 전환
- 버튼 상태: `연결 대기` → `연결 중...` → `대화 중`(듣기/말하기 상태가 실시간 전환) → `종료`
- 상태별 시각적 표시 필요 (예: 마이크 아이콘 색상 변화 — 대기(회색) / 듣는중(초록) / AI 응답중(파랑))

### 1.2 실시간 상태 인디케이터

- "듣고 있어요" / "생각 중이에요" / "말하고 있어요" 상태를 짧은 텍스트+아이콘으로 표시
- 대화 중 자막처럼 인식된 텍스트와 AI 응답 텍스트를 채팅창에도 함께 기록 (음성만으로 끝나지 않고, 텍스트 채팅 히스토리에도 동일하게 쌓이도록 — 대시보드/이력과 일관성 유지)

### 1.3 종료/에러 상태

- 연결 실패 시 "음성 연결에 실패했어요, 텍스트로 계속할까요?" 안내 후 즉시 텍스트 입력창 활성화
- 세션 중 끊김 발생 시 자동 재연결 시도 1회 → 실패하면 텍스트 모드로 자동 전환

---

## 2. 연결 흐름

```
[사용자] 음성 모드 버튼 클릭
   ↓
[프론트] 백엔드에 POST /api/realtime/session 요청 → ephemeral key 수신
   ↓
[프론트] 받은 key로 OpenAI Realtime API에 WebRTC 연결 생성
   ↓
[프론트] 마이크 오디오 트랙을 연결에 추가 (getUserMedia)
   ↓
[프론트] 세션 설정(session.update)으로 tool 목록·한국어·시스템 지침 전달
   ↓
[대화 시작] 사용자 발화 → OpenAI가 실시간으로 인식·응답 생성
   ↓
[OpenAI가 원격 오디오 트랙으로 음성 응답 스트리밍] → 브라우저가 자동 재생
```

---

## 3. Function-call(도구 호출) 이벤트 처리 — 핵심 로직

Realtime API는 대화 중 모델이 "장비를 제어해야겠다"고 판단하면, 오디오가 아니라 **데이터 채널로 function-call 이벤트**를 보냅니다. 이걸 처리하는 흐름이 이번 기능에서 가장 중요한 부분입니다.

```
[OpenAI] function-call 이벤트 수신
   (예: { name: "control_device", arguments: { greenhouse_id: 2, device: "window", action: "open" } })
   ↓
[프론트] 이 이벤트를 그대로 백엔드 POST /api/tools/execute 로 전달
   ↓
[백엔드] 실제 farm_state 변경 후 결과 반환
   ↓
[프론트] 결과를 다시 OpenAI 세션에 주입
   (conversation.item.create 로 function_call_output 전송)
   ↓
[프론트] response.create 호출 → OpenAI가 결과를 반영해 음성으로 응답 계속
```

- 이 흐름이 끊기면 "장비는 실제로 제어됐는데 AI가 그걸 말로 설명 안 함" 같은 불일치가 생길 수 있으므로, function-call 이벤트 수신 → 백엔드 호출 → 결과 재주입까지의 순서를 정확히 지켜야 함
- 백엔드 호출이 실패하는 경우도 function_call_output에 실패 사실을 담아 보내서, AI가 "제어에 실패했어요"라고 말하도록 처리

---

## 4. 세션 설정 (session.update) 시 전달할 내용

- 사용 언어: 한국어
- 시스템 지침: 기존 Claude system prompt와 **톤·판단 기준을 동일하게 맞춰야 함** (작물 생리 기반 설명, 재질문, 중복명령 확인 등 — 텍스트 채팅과 음성 채팅이 다른 성격의 AI처럼 느껴지지 않도록)
- tool 목록: `control_device`, `query_data` (OpenAI 형식 스키마로 별도 작성, 백엔드 문서 참고)

---

## 5. 오디오 재생/입력 처리

- 마이크 권한 요청 (`getUserMedia`) — 최초 진입 시 권한 거부 대응 필요
- 원격 오디오 트랙 자동 재생 (`<audio>` 엘리먼트에 스트림 연결)
- 음소거(mute) 버튼 — 대화 중 일시적으로 마이크를 끄고 싶을 때
- 발표 환경에서는 스피커/마이크 에코 문제가 생길 수 있어 **헤드셋 사용을 전제로 시연 준비** 권장

---

## 6. 에러/재연결 처리

| 상황 | 처리 |
|---|---|
| ephemeral key 발급 실패 | "음성 기능을 시작할 수 없어요" 안내, 텍스트 모드 유지 |
| WebRTC 연결 실패 | 1회 재시도 → 실패 시 텍스트 모드로 자동 전환 |
| 세션 중 연결 끊김 | 재연결 시도, 실패 시 "다시 연결해주세요" 버튼 노출 |
| function-call 브릿지 실패 (백엔드 응답 없음) | 타임아웃 설정 후 실패로 간주, AI에게 실패로 알림 |
| 마이크 권한 거부 | 텍스트 입력으로 안내 |

---

## 7. 데모 시연 관점 체크

- 발표장 네트워크(특히 WebRTC는 방화벽/네트워크 환경에 민감) **사전 반드시 현장 테스트**
- 헤드셋/마이크 준비 (스피커 사용 시 에코로 인식 오류 가능성 높음)
- 음성 모드 실패 시 텍스트로 즉시 전환하는 리허설 포함 (심사 중 실패해도 자연스럽게 넘어가도록)
- 짧고 명확한 시연 문장 준비 (예: "2번 온실 창문 열어줘", "지금 상태 어때?")
